# WireGuardSampleApp — macOS NetworkExtension reference host

A minimal SwiftUI host app + `NEPacketTunnelProvider` extension that
demonstrates how to drive the `WireGuardCore.xcframework` from a real
macOS NetworkExtension target. Bundle IDs:

| Component         | Bundle ID                |
|-------------------|--------------------------|
| Host app          | `com.change.wg`          |
| Tunnel extension  | `com.change.wg.tunnel`   |

The host app provides a config-text editor, Save / Connect / Disconnect
buttons, a live UAPI status pane that polls `sendProviderMessage("get=1\n\n")`
every 2 seconds, and a "UAPI demos" menu that exercises every wg(8) SET
operation (roam endpoint, change keepalive, add peer, remove peer).

## What's in the box

```
WireGuardSampleApp/
├── project.yml                    ← canonical xcodegen spec; edit this
├── WireGuardSampleApp.xcodeproj/   ← generated by `xcodegen generate`
├── WireGuardSampleApp/             ← host app target
│   ├── WireGuardSampleAppApp.swift  ← @main, opens ContentView
│   ├── ContentView.swift            ← SwiftUI two-pane shell
│   ├── TunnelManager.swift          ← NETunnelProviderManager wrapper
│   ├── Info.plist
│   └── WireGuardSampleApp.entitlements
└── WireGuardTunnelExtension/       ← appex target
    ├── Info.plist
    └── WireGuardTunnelExtension.entitlements
```

The actual `PacketTunnelProvider` source lives at
`../NetworkExtension/Sources/PacketTunnelProvider.swift` (one canonical
copy in the repo, referenced from the extension target via relative path
in `project.yml`).

The C library that the extension links against lives at
`../build/xcframework/WireGuardCore.xcframework` and is built by
`make xcframework` in the repo root.

## One-time setup

```bash
brew install xcodegen          # if not already installed
```

## Build steps

From the **repo root**:

```bash
make xcframework               # builds the universal arm64+x86_64 framework
cd WireGuardSampleApp
xcodegen generate              # writes WireGuardSampleApp.xcodeproj
open WireGuardSampleApp.xcodeproj
```

In Xcode:

1. **Set your Development Team** on both targets:
   - Select the project in the navigator
   - For each target (`WireGuardSampleApp` and `WireGuardTunnelExtension`):
     - Signing & Capabilities tab
     - Team: pick your free Personal team or paid Developer ID
2. **Press ⌘R** to build and run.

You can also build entirely from the command line if you only want to
verify the project compiles:

```bash
xcodebuild \
  -project WireGuardSampleApp.xcodeproj \
  -scheme WireGuardSampleApp \
  -configuration Debug \
  -sdk macosx \
  CODE_SIGN_IDENTITY="" \
  CODE_SIGNING_REQUIRED=NO \
  CODE_SIGNING_ALLOWED=NO \
  build
```

(No code signing needed for a build-only check; you only need to sign
when you actually want to run the app, because NetworkExtension
entitlements require a real provisioning profile.)

## Running the app

When you press **Save config** for the first time, macOS pops a dialog
asking for permission to add a VPN configuration — click **Allow** and
authenticate. The new config shows up in System Settings → Network →
VPN as "WireGuard Sample".

When you press **Connect**, the system launches the
`WireGuardTunnelExtension.appex` process, which calls into the
`wg_session` C library and starts the tunnel. The status pill in the
top right turns green and the right-hand pane starts showing live
`wg show`-format text every 2 seconds:

```
private_key=none
listen_port=0
fwmark=0
public_key=5d71736e30e59f4db2fda59fa36445559fdfa22302fcd128f502970dd93d4979
preshared_key=00000000...0000
endpoint=172.16.203.128:51820
last_handshake_time_sec=1734209123
last_handshake_time_nsec=0
tx_bytes=412
rx_bytes=412
persistent_keepalive_interval=25
allowed_ip=0.0.0.0/0
protocol_version=1
errno=0
```

The "UAPI demos" menu lets you fire pre-baked SET requests against the
running session:

- **Roam endpoint to 192.0.2.5:51820** — `wg_session_set_uapi` with
  `endpoint=...`
- **Set persistent_keepalive=15** — same, with the keepalive field
- **Add demo peer** — sends `public_key=` for an unknown pubkey, which
  triggers the runtime peer-add path with endpoint + allowed_ip +
  keepalive
- **Remove demo peer** — `remove=true` after the demo peer's pubkey,
  which tombstones the slot

After each SET demo the response from the extension is shown at the
top of the right pane.

## How the pieces fit together

```
┌──────────────── host app (com.change.wg) ─────────────────┐
│                                                            │
│  ContentView (SwiftUI)                                     │
│      ↓                                                     │
│  TunnelManager  ─────► NETunnelProviderManager             │
│      │                       │                             │
│      │                       │ install / save / start     │
│      │                       ▼                             │
│      │             system VPN preferences                  │
│      │                       │                             │
│      │                       │ launches                    │
│      │                       ▼                             │
│      │         ┌── extension (com.change.wg.tunnel) ───┐   │
│      │         │                                         │  │
│      │         │  PacketTunnelProvider                   │  │
│      │         │     ↓                                   │  │
│      │         │  wg_session_t (C, no I/O)               │  │
│      │         │     ↑       ↓       ↑                   │  │
│      │         │     │       │       └─ NEPacketTunnelFlow │
│      │         │     │       └────────── NWUDPSession    │  │
│      │         │     └─────── DispatchSourceTimer @1Hz    │  │
│      └────►    │     ↑                                   │  │
│  sendProvider  │     │                                   │  │
│   Message      │  handleAppMessage("get=1") / ("set=1")  │  │
│       ◄────────┤     ↑                                   │  │
│                │  UAPI text response                     │  │
│                └─────────────────────────────────────────┘  │
└────────────────────────────────────────────────────────────┘
```

The host app and the extension never share memory directly; everything
goes through `NETunnelProviderSession.sendProviderMessage(_:)`. The
wire format on that channel is the canonical WireGuard cross-platform
UAPI text protocol — exactly what `wg(8)` speaks to a unix socket on
other platforms.

## Editing

`project.yml` is the source of truth. After any edit, run
`xcodegen generate` to write a fresh `project.pbxproj`. **Do not edit
`project.pbxproj` by hand** — the next regenerate will blow your
changes away.

If you change Swift code in `WireGuardSampleApp/` or
`WireGuardTunnelExtension/`, you do not need to regenerate; the
existing project just picks up the new files via path-based references
on the next build.

If you add a new C function to `wg_session.h`, you need to:

1. `cd ..` (back to repo root)
2. `make xcframework` (rebuilds the framework with the new symbol)
3. Build the Xcode project — Swift will pick up the new symbol via
   the framework's module map automatically.
